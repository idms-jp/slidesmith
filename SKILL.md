# SlideSmith — テキストからプロ品質スライドを量産する

## このスキルの思想（最重要）

**AIに毎回レイアウトを考えさせない。** デザインは `core/base.css`（12レイアウトの骨格）と
`themes/*.css`（世界観）に完全固定されている。あなた（Claude）の仕事は
**台本を読み、レイアウトを選び、文章を流し込むことだけ**。
これが「修正ゼロで毎回プロ品質」を実現する唯一の方法である。

## ワークフロー

### STEP 0. 前提確認（初回のみ）
```bash
# このスキルのルートディレクトリで
npm install        # playwright-core が入る（Chrome がローカルに必要）
```

### STEP 1. 要件を確認する
ユーザーから聞くこと（不明なら聞く。台本から自明なら聞かない）:
1. **台本**（txt / md / チャット貼り付け。何でもよい）
2. **テーマ**（下表から選ぶ。指定がなければ内容から推薦して確認）
3. **ブランド表記**（左下に入る名前。例: "SNS LAB"。不要なら省略可）

| テーマ | 世界観 | 向いている用途 |
|--------|--------|--------------|
| corporate | 誠実・上質・ネイビー×ティール | 提案書・会社説明・セミナー |
| pop | 楽しい・ビビッド・丸ゴ | SNS講座・イベント・キャンペーン |
| elegant | 静寂・和モダン・明朝×金 | 高級店・旅館・ハイブランド |
| minimal | 削ぎ落とし・白黒・大胆タイポ | デザイン・建築・ポートフォリオ |
| tech | 未来・ダーク・ネオン発光 | SaaS・AI・スタートアップピッチ |
| warm | 木漏れ日・生成り・低彩度 | カフェ・食・ワークショップ |
| feminine | 大人可愛い・くすみピンク×金 | 美容・女性向け講座・サロン |
| retro | 昭和レトロ・純喫茶・臙脂×芥子 | 商店街・個人店・イベント |

**どれも合わない場合**: `themes/THEME_GUIDE.md` のレシピで新テーマを作る（30分で作れる。
既存テーマの複製から始めると速い）。

### STEP 2. 台本をスライドに分割する
- `core/layouts.md` の「台本→レイアウト対応の目安」に従う
- 鉄則: **coverで始まりclosingで終わる** / 5枚超えならagendaを2枚目に / 同レイアウト3連続禁止
- 1枚 = 1メッセージ。詰め込むくらいなら枚数を増やす
- 分割案（枚数・各枚のレイアウトと見出し）を**先にユーザーへ提示して承認を得る**（10枚超の場合のみ。以下は即実行でよい）

### STEP 3. HTMLを書く
- 置き場所: `decks/<案件名>/01_cover.html` から連番
- **`core/layouts.md` のスニペットを厳守**。独自レイアウトの発明は禁止
- 文字量制限（各スニペットに記載）を厳守。超えそうなら文章を削るか枚数を分ける
- 強調語は `<em>` で囲む（1枚に1〜2箇所。多用禁止）
- **図解優先**: 一番大事な数字は hero-stat、比較は bars、日程は timeline、絞り込みは funnel、優先度は matrix。文章カードの羅列で済ませない
- **レイヤーで奥行きを出す**: layer-texture をほぼ全枚に、ghost（透かし数字）を主役の枚に。「整っているだけ」はシンプル止まり＝失敗（layouts.md「プロ級の鉄則」参照）

### STEP 4. レンダリング & 自動QC
```bash
node scripts/render.mjs decks/<案件名>          # PNG（3840x2160）
node scripts/render.mjs decks/<案件名> --pdf    # PNG + 結合PDF
```
- QC違反（はみ出し・文字あふれ）が出たら該当スライドの文字量を削って再実行
- 「ALL CLEAN」が出るまで納品しない

### STEP 5. 目視レビュー（必須）
- `out/*.png` を Read ツールで開き、**全枚を自分の目で確認**する:
  - 装飾と文字の重なり / 不自然な折り返し / 空白のバランス / 世界観の一貫性
- 問題があれば直して STEP 4 からやり直し
- 完了後、ユーザーにPNGを見せて確認を取る

## 画像・イラストを使う（Phase 3）

### AI写真生成 — 3つのルート（上から順に検討）

| ルート | 費用 | 品質 | 手間 |
|--------|------|------|------|
| ① `--free`（Pollinations.ai） | **0円** | 中〜高（1024px上限。全面背景＋スクリムなら十分） | ゼロ。キー不要で即使える |
| ② Geminiアプリ/ChatGPTで手動生成 | サブスク内 | 高 | 手動（アプリで生成→保存→assets/に置く） |
| ③ Gemini API（要課金設定） | 約6円/枚 | 最高（2K・画風安定） | キー＋課金設定 |

```bash
# ① 無料ルート（まずこれ）
node scripts/genimg.mjs "<日本語で場面を描写>" decks/<案件>/assets/xxx.png --style warm --ar 16:9 --free
```

### ③ Gemini APIルート（1枚約6円/$0.039）
```bash
# 初回のみ: セットアップ（ブラウザでキー発行ページが開く → 非表示の入力欄にキーを貼り付け）
node scripts/setup-gemini.mjs
# ↑ 入力欄は画面にもシェル履歴にも残らない安全方式。引数渡しは履歴に残るため非推奨

# 生成（--style はテーマ名。テーマに合う画風が自動で乗る）
node scripts/genimg.mjs "<日本語で場面を描写>" decks/<案件>/assets/cover.png --style warm --ar 16:9
```
- キー未設定でユーザーが写真を要望したら: `node scripts/setup-gemini.mjs` を実行し、表示されるガイドに沿って**ユーザーに発行を案内**する。キーは**チャットに貼らせない**——ユーザー自身のターミナル（Claude Codeなら `!` プレフィックス）で `node scripts/setup-gemini.mjs` を実行してもらい、非表示プロンプトに貼り付けさせる
- 生成した写真は photo-hero の背景・media・polaroid・breakout（透過が要る場合は「background transparent, isolated on white」を指示）に使う
- 人物の顔アップは崩れやすい。風景・物・後ろ姿・手元が安定

### イラスト（無料・無限）
`core/illustrations.md` に線画アイコン24種＋スポットイラスト6種。badgeやmedia-boxにコピペで使える

### 立体感・斜め文字
`.depth-1/2` `.tilt-l/r` `.polaroid`、photo-hero（斜め帯タイトル）、breakout（円から飛び出す）、kinetic（斜め文字ステートメント）— layouts.md Phase 3セクション参照

## 別サイズ（SNS正方形・縦型）
基本は16:9固定。他サイズが必要なら `.slide` の width/height と render.mjs の viewport を
複製・調整した派生ファイルを作る（安易にbase.cssを直接書き換えない）。

## してはいけないこと
- core/base.css・scripts/ の書き換え（全デッキの品質が壊れる）
- レイアウトの自作・スニペット構造の改変
- 文字量制限の無視（QCに引っかかる前に守る）
- 未レンダリング・未目視での「完成しました」報告
- **写真の主役と文字の重なり**（photo-hero等）。生成プロンプトで被写体位置を必ず指定し（テキストの反対側＋余白）、目視で衝突ゼロを確認してから納品（layouts.md 18の構図ルール参照）
